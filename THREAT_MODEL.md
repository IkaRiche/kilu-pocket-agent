# 👾 Threat Model

This Threat Model outlines the defensive posture of the KiLu Pocket Agent against standard adversarial actions. The model utilizes the STRIDE framework.

## Trust Boundaries
1. **The Human User:** Assumed to be benign but susceptible to social engineering.
2. **The Approver Device:** Assumed secure. Protected by Android OS sandboxing, biometric authentication, and lock screen security.
3. **The Hub Device:** Assumed potentially vulnerable. Operates in untrusted environments (e.g., public Wi-Fi) and executes remote tasks.
4. **The Cloud Control Plane (CCP):** Assumed secure, operating as the source of truth for routing and validation.
5. **The Target Application (Web):** Assumed hostile or uncooperative.

## STRIDE Analysis

### 1. Spoofing
**Threat:** An attacker attempts to submit execution results as a legitimate Hub, or approve a plan as a legitimate Approver.
**Mitigation:** 
* Approver actions require an Ed25519 signature generated by the private key securely stored on the device.
* Hubs require a time-limited, opaque session token provisioned dynamically during the QR pairing process. The Hub possesses no standing privileges.

### 2. Tampering
**Threat:** An attacker intercepts the network traffic between the Agent and the CCP to modify a `Plan` or a `SubmitResultReq`.
**Mitigation:**
* All communication is strictly enforced over TLS 1.3 (`https://`).
* Payloads are cryptographically signed. Any modification in transit invalidates the signature at the destination.

### 3. Repudiation
**Threat:** A user or rogue node denies having authorized a specific action.
**Mitigation:** 
* The CCP maintains a strict, append-only log of every `WindowGrant` submitted, including the user's explicit cryptographic signature authorizing the boundaries.

### 4. Information Disclosure
**Threat:** Task data, extraction results, or session tokens are leaked from the device.
**Mitigation:**
* At rest: Sensitive data (tokens, keys) is stored in Android's `EncryptedSharedPreferences` utilizing AndroidX Security Crypto (AES-256-GCM).
* In transit: TLS 1.3 interception prevents network eavesdropping.

### 5. Denial of Service (DoS)
**Threat:** An attacker floods the Hub with infinite tasks or large payloads, exhausting device battery or memory.
**Mitigation:**
* The Hub polls the CCP at regulated intervals. 
* The CCP enforces aggressive rate limiting, task size limits (`schema maxLength`), and timeout thresholds before serving tasks to the Edge.

### 6. Elevation of Privilege
**Threat:** The execution payload (a malicious skill) breaks out of the Hub's WebView sandbox to execute arbitrary code on the Android host.
**Mitigation:**
* The Hub executes tasks strictly within the isolated Android `WebView` environment. No native code reflection or host-os bridges are exposed to the executing JS context.
* Critical boundaries (domains, methods) are enforced external to the WebView by the CCP egress.
